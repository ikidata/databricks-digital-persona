# Turn your Databricks footprint into a virtual version of you

![digital_image](https://github.com/ikidata/databricks-digital-persona/blob/main/.images/feature_image_video_v2.gif?raw=true)

> **What if an agent could think and act exactly like you?**

___


Every Databricks user leaves a trail of activity in system tables (queries, jobs, services, costs, login patterns). But that data sits unused too often. This solution reads those logs and transforms them into something meaningful, a virtual version of you:

1. **Mirror**  — See how you actually use the platform, not how you think you do
2. **Clone** — A Digital Twin that thinks, answers, and operates like you
3. **Persist** — Your expertise, available even when you're not

---

## The goal of the solution

This project demonstrates how it's possible to create virtual personas based on system log data. By analysing your Databricks workspace activity — queries, jobs, services, costs, schedules — the app builds a rich behavioural profile and converts it into your own **Digital Twin**: an AI agent you can talk to that knows your work habits, expertise and personality.

Built as a [Databricks App](https://docs.databricks.com/aws/en/dev-tools/databricks-apps/) using Flask. Runs entirely inside your workspace using **on-behalf-of-user (OBO) authentication** — your data stays yours.

## How It Works — Step by Step

```
  System Tables          Analysis Engine         Virtual Persona         Digital Twin
  ─────────────          ───────────────         ───────────────         ────────────
  audit logs        ──>  15-step data       ──>  archetype scores   ──>  AI system prompt
  query history     ──>  collection from    ──>  achievement badges ──>  generated by Claude
  job runs          ──>  8+ system tables   ──>  work patterns      ──>  that you can chat
  table lineage     ──>  (~5-10 minutes)    ──>  KPIs & expertise   ──>  with directly
  billing/costs     ──>                     ──>  reliability score
  compute usage
  serving endpoints
  git activity
```

### Step 1 — Check Access

Verify your permissions on the required system tables. The app tests each table and shows which ones are accessible.

### Step 2 — Run Analysis

The analysis engine collects 90 days of your activity from 8+ Databricks system tables:

| Data Source | What It Reveals |
|---|---|
| `system.access.audit` | Services used, work patterns, dashboard activity, GenAI usage |
| `system.query.history` | SQL behaviour, query complexity, catalogs touched |
| `system.lakeflow.jobs` | Pipeline reliability, scheduling habits |
| `system.access.table_lineage` | Tables you read and write, data ownership |
| `system.billing.usage` | DBU spend, compute efficiency |
| `system.compute.clusters` | Cluster preferences, autoscaling habits |
| `system.serving.endpoint_usage` | Model serving & AI endpoint usage |
| `system.access.audit` (GenAI) | Genie conversations, agent framework activity |

This runs 15 collection steps in the background (~5-10 minutes). Progress is streamed live to the browser.

### Step 3 — View Your Persona

The Persona page shows your full profile:

- **Archetype classification** — scored against 6 behavioural archetypes
- **Achievement badges** — 20+ automatically earned from your data
- **Canvas visualisations** — animated waveforms, radial clocks, heatmaps
- **KPIs and statistics** — query counts, job reliability, cost efficiency

### Step 4 — Meet Your Digital Twin

Navigate to the **Digital Version** page. Claude writes a vivid first-person system prompt based on your behavioural profile — then you can chat with it directly. The prompt captures your personality, habits, and expertise, not just raw numbers.

## Quick Start

### Prerequisites

- Databricks workspace with **Unity Catalog enabled**
- `SELECT` access on `system.*` tables
- A **SQL Warehouse** (Serverless or Pro)
- The **[Databricks CLI](https://docs.databricks.com/dev-tools/cli/index.html)** v0.250+
- The model endpoint **`databricks-claude-sonnet-4-6`** available in your workspace

### 1. Update `databricks.yml`

Before deploying, edit `databricks.yml` with your actual workspace details:

```yaml
resources:
  apps:
    digital_persona:
      resources:
        - name: "starter-warehouse"
          sql_warehouse:
            id: "your_warehouse_id"     # ← your SQL warehouse ID
            permission: "CAN_USE"
        - name: "llm-endpoint"
          serving_endpoint:
            name: "databricks-claude-sonnet-4-6"  # ← your model endpoint
            permission: "CAN_QUERY"

targets:
  dev:
    workspace:
      host: https://your-workspace.azuredatabricks.net  # ← your workspace URL
```

The warehouse ID is automatically passed to the app at runtime via `valueFrom: "starter-warehouse"` in `app.yaml` — no need to hardcode HTTP paths.

### 2. Deploy to Databricks

```bash
databricks bundle validate
databricks bundle deploy
```

If the bundle deploy doesn't start the app automatically, deploy directly:

```bash
databricks apps deploy digital-persona --source-code-path /Workspace/Shared/digital-persona-app/dev/files
```

Or deploy and update the source code manually from the Databricks workspace UI under **Apps** → **digital-persona**.

### 3. Use the App

1. **Check Access** — verify your system table permissions
2. **Run Analysis** — collects data in ~15 steps (runs in background, 5-10 min)
3. **View Persona** — explore your archetype, badges, KPIs, and charts
4. **Digital Version** — get your AI-generated Digital Twin system prompt and chat with it

## Local Development

For local development you need a `config.yaml` with your Databricks credentials (this file is git-ignored and never committed):

```bash
cp config.yaml.example config.yaml
```

Edit `config.yaml` with your workspace details:

```yaml
databricks:
  host: "https://your-workspace.azuredatabricks.net"
  auth_method: "pat"                  # "pat" is simplest for local dev
  token: "dapi_your_personal_access_token"
  sql_warehouse_http_path: "/sql/1.0/warehouses/your_warehouse_id"

chat:
  model_serving_endpoint: "databricks-claude-sonnet-4-6"
  max_tokens: 1024
```

> **Note:** `config.yaml` is only needed for local development. When deployed as a Databricks App, authentication and warehouse configuration come from the asset bundle resources and OBO tokens automatically.

Then run:

```bash
pip install -r requirements.txt
python run_locally.py --email you@yourcompany.com
# → http://localhost:8000
```

The `--email` argument is **required** — it sets the Databricks user identity used to query your activity data from system tables. Use the same email you log in to Databricks with.

The local server auto-reloads on file changes. All analysis data is in-memory and resets on server restart.

## Testing

The project includes **90 tests** across 4 test modules:

```bash
pytest tests/ -v                # run all tests
pytest tests/test_scoring.py    # archetype scoring & classification (28 tests)
pytest tests/test_security.py   # security checks — secrets, XSS, headers (18 tests)
pytest tests/test_store.py      # data store — CRUD, isolation, threads (17 tests)
pytest tests/test_routes.py     # Flask routes — pages, API, validation (27 tests)
```

| Test Module | What It Covers |
|---|---|
| `test_scoring.py` | Archetype scoring functions, sigmoid math, classification distribution, metadata |
| `test_security.py` | No hardcoded secrets, config safety, security headers, XSS prevention, token handling, SQL injection patterns, debug mode, log safety |
| `test_store.py` | Persona CRUD, user data isolation, analysis status, LLM prompt storage, thread safety |
| `test_routes.py` | All page routes return 200, digital version gate (blocks without data), API validation, response security headers |

All tests run **without a Databricks connection**. The route tests mock the Databricks SDK so they work in any environment.

### Security Scan

A standalone security scanner is also available:

```bash
python security_scan.py
```

This checks for hardcoded secrets, SQL injection risks, XSS patterns, debug mode, CSRF, rate limiting, token handling, error leakage, and dependency vulnerabilities.

## How the Analysis Engine Works

The analysis engine (`analyzer.py`) runs 15 collection steps in sequence:

1. Resolve user identity (SCIM API)
2. Overall activity summary (audit log)
3. Service breakdown & hourly/weekly patterns
4. SQL query history (complexity, catalogs, languages)
5. Pipeline/job reliability (lakeflow.jobs)
6. Data lineage (tables read/written)
7. Collaboration (downstream consumers)
8. Compute usage (clusters)
9. GenAI & LLM usage (Genie, agentFramework, serving endpoints)
10. Git activity
11. Cost & DBU efficiency
12. Dashboard activity (sessions, engagement depth, dashboard names)
13. Engagement timeline & streaks
14. Workspace comparison (percentile ranking)
15. Archetype classification & badge scoring

Progress is streamed live to the browser via SSE (Server-Sent Events).

## Authentication & Security

### In Production (Databricks Apps)

The app uses **on-behalf-of-user (OBO)** authentication:
1. Databricks Apps reverse proxy injects the user's token as `X-Forwarded-Access-Token`
2. Every SQL query runs under **the logged-in user's own permissions**
3. No shared credentials — each user only sees their own data

### Required Permissions for the App Service Principal

| Resource | Permission |
|---|---|
| SQL Warehouse | CAN_USE |
| `databricks-claude-sonnet-4-6` serving endpoint | CAN_QUERY |

### Required `user_api_scopes` (consent prompt on first use)

| Scope | Purpose |
|---|---|
| `sql` | Run queries via the SQL warehouse |
| `serving.serving-endpoints` | Chat with the Digital Twin |
| `catalog.catalogs:read` | Read Unity Catalog metadata |
| `catalog.schemas:read` | Read Unity Catalog schemas |
| `catalog.tables:read` | Read system tables |

### Security Properties

- `config.yaml` is git-ignored — local credentials never reach the repo
- Flask secret key uses `secrets.token_hex(32)` fallback if not set via env var
- Security headers on every response: `X-Content-Type-Options`, `X-Frame-Options`, `CSP`, `Referrer-Policy`
- All queries are scoped to the authenticated user's email
- Workspace comparison stats show only anonymous averages, never individual data
- 90 automated tests including dedicated security test suite
